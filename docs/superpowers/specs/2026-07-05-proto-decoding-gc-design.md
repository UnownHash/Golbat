# Proto Decoding GC Reduction — Opaque API + Lazy Decoding

**Date:** 2026-07-05
**Status:** Approved design, pending implementation plan
**Branch:** `worktree-proto-opaque-gc` (worktree off `main`; `perf/eviction-lock-contention` untouched)

## Problem

Golbat decodes every raw protobuf from game clients into full `pogo` message
trees. Each GMO response allocates hundreds of heap objects (one per
submessage, plus strings and repeated-field slices), all garbage within
seconds. This drives GC CPU cost and bounds decode throughput.

**Success metrics** (in priority order):
1. **GC CPU share** — % of CPU in GC mark/scan/assist, from prod CPU profiles.
2. **Decode throughput/latency** — time per raw packet; fewer shed packets
   (`golbat_raw_packets_shed_total`) under load.

## Current state (verified)

- `pogo/vbase.pb.go` (484k lines) generated by **protoc-gen-go v1.31** (open
  struct API). Runtime is `google.golang.org/protobuf v1.36.11`, which fully
  supports the Opaque API and lazy decoding.
- Hot messages (`PokemonProto` — 78 fields, `WildPokemonProto`,
  `PokemonFortProto`) are **pure proto3 implicit presence: zero pointer-boxed
  scalars**. The Opaque API's "presence pointers → bitfields" win does not
  apply; the allocation cost is the per-submessage objects themselves.
  Therefore **lazy decoding is the lever; opaque is its prerequisite**.
- Decoded protos escape the decode call in two ways:
  - Transient (seconds): `decoder/main.go` Raw batch structs
    (`*pogo.PokemonFortProto`, `*pogo.WildPokemonProto`, …).
  - Long-lived (hours): `decoder/weather_consensus.go` retains
    `*pogo.ClientWeatherProto` per S2 cell.
- Only **13 construction sites** of pogo protos in non-test code, nearly all
  empty `&pogo.X{}` literals (legal under opaque). Migration cost is
  mechanical field-read rewrites across ~31 files.
- No decode benchmarks or captured payload fixtures exist.
- `/debug/pprof/{allocs,heap,profile}` endpoints exist behind auth
  (`main.go:357-372`).

### Prior art: PR #323 (`len-optimize`, closed)

Prototyped opaque codegen (`default_api_level=API_OPAQUE`, dual
`vbase_protoopaque.pb.go` behind `//go:build protoopaque`), a lazy-annotation
script (`scripts/add_lazy_proto.py` — annotates `[lazy = true]` on submessage
fields whose getters are never called), hand-migrated call sites (~1,100
lines), plus jsonv2 and greenteagc experiments. Prod-tested with
**inconclusive results (no baseline captured)**; closed as outdated after main
diverged. Its two failure causes — no measurement, unrepeatable hand
migration — are directly addressed below.

## Design

Three phases, each independently mergeable, each gated on measurement.

### Phase 0 — Prove the methodology standalone (no Golbat integration)

Golbat's dominant decode pattern is decode → read fields → drop for GC. That
is simulable at volume without integrating anything, so the decode
methodology is proven (or killed) before any of the ~31-file migration
happens.

1. **Payload corpus**: debug-gated capture hook on the raw path writing N raw
   protobuf payloads per method type (default N=25; GMO, Encounter,
   FortDetails, …) from prod to disk. Fixtures live in a gitignored local directory (payloads may
   contain account-adjacent data; not committed to the public repo). This
   hook is the only Golbat change in Phase 0 (instrumentation-only,
   debug-gated).
2. **Standalone decode harness** — a separate Go module (`protobench/`, own
   `go.mod`, in-repo). Separate module is mandatory, not stylistic: a second
   generated copy of `vbase.proto` in the same binary as `golbat/pogo`
   panics the global proto registry on duplicate file registration. The
   module contains:
   - Its own pogo package generated at `API_HYBRID` **with lazy annotations
     already applied**, giving three configurations from one package:
     **(a)** default build = open struct layout (current-Golbat baseline);
     **(b)** `-tags protoopaque` = opaque with lazy active;
     **(c)** `-tags protoopaque` + `proto.UnmarshalOptions{NoLazyDecoding:
     true}` = opaque without lazy. Hybrid getters compile identically in all
     three modes, so the harness code is written once.
   - **Access simulation**: per-method reader functions that call exactly
     the getter set Golbat's decode path uses (derived from the same
     used-getter analysis `add_lazy_proto.py` performs), mimicking
     decode → read → drop.
   - **Volume runs**: configurable worker count (default 96, mirroring
     `raw_processing_concurrency`) decoding the corpus in a sustained loop;
     reports decodes/s, ns/op, B/op, allocs/op, GC CPU share
     (`runtime/metrics` `/cpu/classes/gc/...` deltas), heap in-use, and
     pause distribution. An optional pointer-dense heap ballast flag
     approximates Golbat's large live caches, since GC mark cost scales
     with live heap.
3. **Prod baseline**: alloc/heap/CPU profiles at peak; record GC CPU share and
   decode-duration distribution as the before-picture for the later canary
   phases.

**Phase 0 exit gate (decides whether Phase 1 happens at all)**: configuration
(b) or (c) must beat (a) at volume on allocation rate and harness GC CPU
share (provisional target ≥20% allocs/op reduction on the GMO workload,
firmed up once the corpus exists). The harness proves *relative* methodology
gain; absolute prod numbers still come from the canary phases, since prod GC
shares the heap with caches, R-trees, and write-behind queues.

### Phase 1 — Opaque API migration (starts only if the Phase 0 gate passes)

**1a. Codegen + call-site rewrite (default behavior unchanged):**
- Port `updateproto.sh` + Makefile target from `len-optimize`, updated to:
  pinned protoc-gen-go ≥ v1.36, proto source path via env var (default
  `~/dev/ProtoMirror/vbase.proto`), `default_api_level=API_HYBRID`.
- Hybrid emits `vbase.pb.go` + `vbase_protoopaque.pb.go`
  (`//go:build protoopaque`). Default builds keep the open struct layout;
  `-tags protoopaque` builds the opaque layout. CI tests **both** modes.
- Migrate call sites with Google's **open2opaque** tool (not by hand); manual
  cleanup expected for field-address-taking and `!= nil` → `Has*()` presence
  checks. Repeatable if main moves — reruns are cheap.
- **Differential decode test**: run the Phase-0 corpus through the decode path
  in both build modes; diff resulting entity states and webhook payloads.
  Catches semantic drift (nil-check vs presence semantics) that unit tests
  miss.

**1b. Canary + flip:**
- Canary a `-tags protoopaque` build on prod; compare GC CPU share,
  decode-duration histogram, shed counter, RSS at matched traffic.
- Gate: **neutral-or-better** (Phase 1 is the enabler, not the win).
- On pass: flip codegen to `API_OPAQUE`, delete the dual variant.

### Phase 2 — Lazy decoding (where the win is demanded)

- Extend `scripts/add_lazy_proto.py`: annotate `[lazy = true]` on submessage
  fields (singular **and** repeated — both eligible per the runtime) whose
  getters are never called in Golbat. Getter-name collisions across messages
  err safe (false "used" → fewer annotations). Realistic targets: Niantic
  subtrees Golbat never reads (sponsor/ad/display/event payloads).
- **Retention denylist** (the fix PR #323 lacked): no lazy annotations on any
  field reachable from message types retained long-term. A lazy field keeps
  the deferred subtree's encoded bytes — and the unmarshal input buffer —
  alive until first access; the retained `ClientWeatherProto` would pin
  GMO-sized buffers for hours. Denylist seeded with `ClientWeatherProto`'s
  subtree; implementation includes an audit for other retained roots.
- **Runtime A/B switch**: thread a config flag through the unmarshal calls
  using `proto.UnmarshalOptions{NoLazyDecoding: true}`. Ships defaulted to
  lazy **off**; same binary flips it on canary.
- Note: the Phase-0 harness runs lazy **without** the retention denylist
  (nothing is retained in a decode → read → drop loop); the denylist exists
  purely for Golbat integration, where retained protos would pin buffers.
- Gate: the Phase-0 harness win reproduces in prod — GC CPU share down on
  canary with the flag flipped; `inuse_space` flat (pinned-buffer failure
  mode); decode-duration histogram neutral-or-better.

### Rollout, rollback, operations

- Shipping order: Phase 0 (corpus + standalone harness, **exit gate**) →
  1a → 1b canary → flip → Phase 2 (flag off) → canary A/B → default-on.
  If the Phase 0 gate fails, the effort stops with only the capture hook and
  harness merged — no migration cost incurred.
- Rollback levers: config flag off (lazy); build without tag (opaque,
  pre-flip); revert regen commit (post-flip).
- **One-command regen**: `updateproto.sh` (denylist + lazy script + pinned
  protoc) must stay a single command — FurtiF ships a new `vbase.proto` every
  game version; annotations are regenerated, never hand-maintained.
- Document the Opaque API contributor ergonomics (getters/setters/builders,
  no composite literals with fields) and the regen pipeline in CLAUDE.md.

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| open2opaque semantic drift (presence semantics) | Differential decode test over the corpus in both build modes |
| Lazy pinning of receive buffers via retained protos | Retention denylist; canary watch on `inuse_space` |
| Rebase races with in-flight branches | Regen and rewrite both scripted; rerunning is cheap |
| Lazy access-time overhead where fields *are* read | Unused-fields-only annotation policy; harness access simulation exercises the real getter set |
| Harness not representative of prod GC (live heap, other subsystems) | Pointer-dense ballast option; harness gates on relative gain only; prod canaries remain the absolute gates |
| Upstream vbase.proto churn per game version | One-command pipeline, annotations auto-regenerated |

## Out of scope

- **jsonv2 webhook/API codec** (from PR #323) — separate follow-up.
- **Green Tea GC** — already default on Go 1.26.
- **vtprotobuf** — mutually exclusive with opaque; revisit only if Phase 1/2
  numbers disappoint.
- **Message pooling / receive-buffer pooling** — unsafe with lazy fields
  aliasing input buffers; possible later work for non-lazy paths.
- **Golbat's own `grpc/` protos** — not where the garbage is; may follow the
  same pipeline later.
