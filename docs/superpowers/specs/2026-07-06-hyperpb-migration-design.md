# Hyperpb Decode Engine Migration — Design

**Date:** 2026-07-06
**Status:** Draft for review
**Branch:** `perf/proto-decode-phase0` (PR #378)
**Precedents:** `2026-07-05-proto-decoding-gc-design.md` (Phase 0 — opaque/lazy
gate FAILED), `protobench/RESULTS.md` (engine measurements)

## Problem and goal

Golbat's decode path allocates a full protobuf-go message tree per raw packet
and drops it seconds later; under a production-sized live heap this
allocation rate dominates GC cost. Phase 0 killed the opaque/lazy hypothesis;
the harness then measured hyperpb (buf.build/go/hyperpb: compiled parser,
arena allocation, lazy subtrees, zero-copy) as the decisive winner.

**Goal:** decode client protos with hyperpb behind generated typed accessor
shims, per-method and config-gated, with std as permanent fallback.

**Success metrics** (prod canary, per the Phase 0 baseline):
1. GC CPU share down materially (harness: 5.1% → 1.4% decode-only; 3.2–3.6×
   decode throughput under a 2 GB live-heap ballast).
2. Decode duration / shed-packet rate improved at matched traffic.
3. Zero behavioral drift, proven by shadow comparison before each flip.

Golbat is strictly **read-only** against client protos (verified; hyperpb
panics on mutation), and targets amd64/arm64 (hyperpb's supported set).

## Measured basis (frozen prod corpus; full data in protobench/RESULTS.md)

| engine | decodes/s (GMO) | B/decode | objects | GC CPU |
|---|---|---|---|---|
| std (today) | 34.4k | 222 KB | 1,357 | 5.1% |
| hyperpb+PGO via shims | 57.6k | 49.8 KB | 255 | 1.5% |
| — with 2 GB ballast | 3.2–3.6× std | −78% | −81% | 7.5%→3.7% (GOGC 200) |

Shim-layer overhead vs hand-rolled reflection: ~1.7%. Cross-engine
correctness: identical access-walk digests over all 1,000 corpus payloads.

## Design

### 1. Shim generation (`pogoshim` package)

Port `protobench/cmd/hyperpbgen` into Golbat (`cmd/pogoshimgen`), generalized
in one way: root messages are resolved from the `golbat/pogo` descriptor
registry by full name instead of compile-time references, so the root set is
a flag. Generated output `pogoshim/` (committed, regenerated by the existing
proto-update script):

- One value-type wrapper per message in the transitive closure of every
  inbound response proto Golbat decodes (`GetMapObjectsOutProto`,
  `EncounterOutProto`, `DiskEncounterOutProto`, `FortDetailsOutProto`,
  `GymGetInfoOutProto`, `FortSearchOutProto`, `GetMapFortsOutProto`, and the
  remaining ~15 dispatch cases in decode.go).
- protoc-style getters backed by cached FieldDescriptors: nil-chaining-safe
  zero shims, float32 getters return float32 (arithmetic parity), enums
  return `golbat/pogo` enum types (existing code keeps compiling), repeated
  fields return `List` wrappers with `Len/At/All(iter.Seq)`.
- Maps and oneofs: not generated in v1; the two oneof sites Golbat reads
  (e.g. `PokestopIncidentDisplayProto.MapDisplay` character display) get
  hand-written accessors in a small committed `pogoshim/manual.go`, tested
  against golden payloads.

### 2. Engine runtime (`decoder/protoengine.go` or main package)

- Compiled `hyperpb.MessageType` per root, built at startup after config
  load; stored in `atomic.Pointer`s.
- **PGO warmup:** optional (`proto_engine.pgo = true`): record a profile over
  the first 256 packets per method (or a local capture dir when present),
  `Recompile` in a background goroutine, atomically swap. Measured +4%.
- **Arena lifecycle:** pool of `hyperpb.Shared` per method family. Invariant:
  an arena is freed exactly when its decode call returns — all batch
  processing inside `decode()` is synchronous today and must stay so for
  shim-carrying values. Helper owns the pattern:

  ```go
  func withArena[T any](pool *sync.Pool, ty *hyperpb.MessageType,
      payload []byte, wrap func(protoreflect.Message) T, fn func(T) string) (string, error)
  ```

- **Buffer invariant (zero-copy):** the payload `[]byte` must outlive the
  arena. Today each payload is a fresh allocation owned by `decode()` —
  already satisfied. Ingest-buffer pooling (a measured −14% bytes win) is
  DEFERRED until it can share the arena's release point; the invariant is
  documented at both sites.
- **Engine selection:** config map, default std:

  ```toml
  [proto_engine]
  gmo = "hyperpb"       # std | hyperpb
  encounter = "hyperpb"
  # unlisted methods = std
  shadow_sample_rate = 0.01   # 0 disables shadow comparison
  ```

- **Portability:** hyperpb glue lives behind `//go:build amd64 || arm64`
  with a stub that forces std elsewhere; std remains a permanent fallback,
  not a transition artifact.

### 3. Call-site migration (per method, three waves)

Decode functions change signature from `*pogo.X` to `pogoshim.X`; field
reads become getter calls; repeated-field loops use `.All()`. Entities
already copy values out, so nothing downstream changes.

- **Wave 1 — Encounter family** (`ENCOUNTER`, `DISK_ENCOUNTER`): deep reads,
  modest fan-out, exercises the arena + shim + shadow machinery end to end.
- **Wave 2 — GMO**: the prize (forts, wild/nearby/map pokemon, weather,
  cells, stations). Includes the boundary fixes below.
- **Wave 3 — remaining methods** (fort details, gym info, quests, routes,
  invasions, tappables, …), lowest volume last.

**Boundary fixes (Wave 2 prerequisites):**
1. `weather_consensus.go` stores `*pogo.ClientWeatherProto` for hours →
   replace with a Golbat-owned value struct of the fields it reads
   (condition, display levels, cell id, timestamps). Standalone refactor,
   safe to land before any engine change.
2. `decoder/main.go` Raw batch structs carry shims instead of proto
   pointers; review-assert no consumer stores them beyond the batch call.
3. Audit for any other retained client-proto reference
   (`grep -rn '\*pogo\.' decoder/ | grep -v _test` against struct fields) —
   as of this writing weather consensus is the only long-lived one.

### 4. Shadow comparison (the correctness gate)

Side-effect-free by construction: for a configured sample of packets, decode
with BOTH engines and compare a canonical **digest** of the extracted field
values (the protobench equivalence-walk approach — same field set, same
fold), never running entity processing twice. Mismatches increment
`golbat_proto_shadow_mismatch_total{method}` and log the payload to the
capture dir for offline diffing.

Flip gate per method: mismatches = 0 over ≥3 days at ≥1% sampling on prod
traffic, then set the engine flag; the flag flip is also the rollback lever.

### 5. Operations

- **Proto version bumps** regenerate shims in the same one-command pipeline
  as `vbase.pb.go` (getter surface is compile-checked; drift = build error,
  by design).
- hyperpb pinned by version in go.mod; upgrades go through the protobench
  harness (equivalence + volume run) before Golbat sees them.
- CLAUDE.md documents the engine architecture, the arena/buffer invariants,
  and the shim regen step.

### 6. Bundled quick wins (independent, can land first)

- `DiscardUnknown: true` on all std client-proto unmarshals (+3.7% rate,
  −5.5% objects; Golbat never re-serializes client protos).
- GOGC guidance for prod (200–400 with RAM headroom) in docs — measured
  +72% std decode rate under ballast; complements rather than replaces the
  engine work.

## Testing

- **Unit:** shim getters against golden payloads built with pogo builders
  (synthetic; no corpus dependency in CI).
- **Digest equivalence in CI:** std-vs-hyperpb digest walks over synthetic
  payloads for every migrated method.
- **Perf rig:** protobench + frozen corpus remains the regression harness;
  re-run the engine matrix on hyperpb upgrades and proto bumps.
- **Prod:** shadow counters (gate), then canary GC share / decode-duration /
  shed-rate vs the Phase 0 baseline profiles.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| hyperpb is v0.1.x (API churn, unknown bugs) | Shim layer isolates call sites; version pinned; std engine permanent runtime fallback; shadow gate before every flip |
| Arena outlives payload buffer or vice versa (zero-copy aliasing) | Single `withArena` helper owns the lifecycle; buffer pooling deferred until coupled; invariant documented |
| Retained shim/message escapes its arena | Boundary fixes (weather consensus first); review checklist; Raw-struct audit |
| Digest walk diverges from real field usage | Digest generated from the same shim getter surface the decode functions use; float32 parity rule from Phase 0 baked into the generator |
| Non-amd64/arm64 builds break | Build-tagged glue + std stub |
| Per-game-version regen churn | One-command scripted pipeline; compile-checked surface |

## Out of scope

- Webhook/API JSON marshaling (jsonv2 — separate follow-up once decode no
  longer dominates profiles).
- Ingest buffer pooling (deferred until arena-coupled release exists).
- vtprotobuf (`vtpool`) — retained in the harness as the measured fallback
  engine should hyperpb prove unviable; not built into Golbat now.
- The opaque API migration — dead per the Phase 0 gate.
